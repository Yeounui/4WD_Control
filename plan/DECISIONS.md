# Decisions

Chronological decision log. Each entry records the choice, the reason, and the
evidence or source. Status terms per `.claude/rules/Edit_Workflow.md`.

## D1 — Architecture adopted from `plan.md` (module-per-driver + central FSM + fixed tick)

The architecture (one driver module per peripheral/algorithm, a central 7-state
FSM, and a fixed 10 ms control tick orchestrating Kalman → PID → S-curve → FSM)
is taken directly from the user's `plan.md`. It is not invented here.

A light benchmark of comparable open-source STM32 robot-car firmware confirms this
is a standard, well-supported shape — adopted as-is, no new structure borrowed:

- [TelecomParistoc/MotorController](https://github.com/TelecomParistoc/MotorController)
  — separates high-level order computation from a PID loop fed by coding-wheel +
  IMU feedback. Confirms the encoder + IMU + per-wheel-PID fixed-tick pattern.
- [SS2867/STM32RobotCar](https://github.com/SS2867/STM32RobotCar) — integrated
  line-tracking / parking / control project organized module-per-capability.
- [sametoguten/STM32-Line-Follower-with-PID](https://github.com/sametoguten/STM32-Line-Follower-with-PID)
  — PID line follower on STM32F103, confirms the centroid-error PID approach.
- [KevinAljas/vl53l1x-stm32-hal-driver](https://gitlab.com/KevinAljas/vl53l1x-stm32-hal-driver)
  and [pololu/vl53l1x-arduino ContinuousMultipleSensors](https://github.com/pololu/vl53l1x-arduino/blob/master/examples/ContinuousMultipleSensors/ContinuousMultipleSensors.ino)
  — confirm the XSHUT-gated, one-at-a-time I2C address-reassignment sequence used
  in [[ARCHITECTURE]] §vl53l1x.

## D2 — MCU is STM32F103RB; `plan.md` "SRAM 20 KB" was correct (D2 reversed)

`plan.md` listed the MCU as `F103RE` but stated `SRAM 20 KB`. These are
inconsistent: F103RE = 64 KB SRAM / 512 KB Flash, while 20 KB SRAM corresponds to
F103R**B** (128 KB Flash).

**Previously concluded (2026-06-23):** the chip was taken to be **F103RE**, treating
the "20 KB" figure as a typo.

**Reversed (2026-06-24):** user confirmed the physical board is the standard
**Nucleo-F103RB** = **STM32F103RBT6** (128 KB Flash / 20 KB SRAM). The original
"20 KB SRAM" figure was accurate; the chip is **F103RB**, not F103RE.

- CubeMX device selection, linker script, and startup file target **STM32F103RB**.
- The no-`malloc` / static-buffer discipline ([[OVERVIEW]] §Constraints) is kept
  regardless of the smaller SRAM — good practice, not a capacity workaround.
- Nucleo-F103RB is a real standard ST board (resolves the prior "no standard
  NUCLEO-F103RE" concern).

## D3 — Debug output reuses USART2 during bench tuning

`plan.md` line 354 proposed debug printing over "UART1 (PA9/PA10) or SWO". All of
these collide with assigned functions:

- USART1 default PA9/PA10 = LR-forward / LF-reverse motor inputs
- USART1 remap PB6/PB7 = I2C1 (SCL/SDA)
- SWO/TRACESWO (PB3) = LF-forward motor input

Decision (user-confirmed, 2026-06-23): during bench bring-up and tuning, before
HC-06 is physically attached, reuse **USART2 (PA2/PA3)** as the debug output.
Phases that depend on a debug path — Phase 3 (yaw via UART) and the Kalman /
straight-line data collection in [[REVIEW]] — run on USART2 with HC-06 detached.
Once HC-06 occupies USART2, on-target debug printing is unavailable; rely on LCD
status and pre-collected data instead.

## D4 — HAL + LL split per AN5044

- **HAL**: motor GPIO, I2C1, ADC1-DMA, USART2-DMA, SysTick.
- **LL**: Hall-sensor and shock-sensor EXTI initialization and IRQ handling.

Mixed HAL/LL in one project remains supported per ST application note AN5044,
and the latency-critical EXTI path remains LL as originally planned. CubeMX
selects the driver per generated initialization function, so `MX_GPIO_Init`
uses LL for all pin initialization while the hand-written motor runtime API
continues to use HAL GPIO calls with HAL `GPIO_PIN_x` masks rather than the
LL-encoded pin macros generated in `main.h`.

## D5 — CubeMX-first, generated by Claude headless; hand-code in USER CODE sections

Each phase first configures the needed peripherals in STM32CubeMX and regenerates
code, then driver/algorithm logic is written by hand inside
`/* USER CODE BEGIN */ … /* USER CODE END */` blocks so CubeMX regeneration does
not overwrite it.

**Ownership (user-confirmed, 2026-06-23):** Claude performs the CubeMX generation
itself — it is *not* a manual user step. STM32CubeMX **6.17.0** at
`/opt/st/stm32cubemx_6.17.0/STM32CubeMX` (bundled JRE `/opt/st/jre`) does headless
CMake generation. **Verified 2026-06-23** by a smoke run of this script
(`STM32CubeMX -q <script>`), which produced `CMakeLists.txt`, `CMakePresets.json`,
`STM32F103XX_FLASH.ld`, and `startup_stm32f103xb.s`:

```
load STM32F103RBTx        # full part string required (not "STM32F103RB")
project name <name>
project toolchain CMake
project path <dir>
project generate
exit
```

The generated `startup_stm32f103xb.s` + `F103XX_FLASH.ld` independently confirm
the xB density (128 KB/20 KB) from §D2.

**Per-phase pattern:** the CubeMX CLI cannot mux pins/peripherals via script —
scripting only loads, sets project metadata, and generates. So Phase 0 creates a
baseline `.ioc` via `load`, and each later phase edits that `.ioc` (plain
key=value text) to add the phase's peripherals, then re-runs the headless
generate. Driver logic stays in `/* USER CODE BEGIN/END */` blocks. The user's
only role in the CubeMX step is the physical hardware gate. This is not full
end-to-end autonomous generation — coding still happens per phase via the
orchestration protocol.

## D6 — Day-based schedule restructured into capability phases

`plan.md`'s 9-day calendar (2025-06-22 ~ 2025-06-30) is reorganized into

## D7 — Drive hardware is a two-input H-bridge shield, not continuous-rotation servos

**Corrected 2026-06-24 from the physical wiring:** each DC motor uses two shield
inputs. The first input is forward, the second is reverse, and both LOW is stop.
Phase 1 therefore uses GPIO direction control and removes TIM1 servo PWM.

| Motor | Shield inputs | MCU GPIO |
|---|---|---|
| Right rear | 21 / 22 | PB8 / PB9 |
| Right front | 17 / 16 | PB4 / PB5 |
| Left rear | 12 / 27 | PA9 / PB10 |
| Left front | 25 / 26 | PB3 / PA10 |

This mapping is the same polarity used by the previously working Smart_Car
firmware. Direction changes clear both inputs before asserting the requested
input. PB3, PB4, PB8, PB9, and PB10 were previously assigned to Hall/shock/XSHUT
signals, so those later-phase signals must be remapped before their phases.
Closed-loop speed PID also needs a separate, verified speed-modulation mechanism;
Phase 1 intentionally implements direction only.

## D8 — Resolve motor-pin conflicts on unused Nucleo header pins

**Decided 2026-06-24:** use PA4 for Hall RR, PA6 for shock EXTI, and
PA1/PA7/PA8 for Front/Left/Right VL53L1X XSHUT. These pins are exposed on the
NUCLEO-F103RB Arduino or ST morpho headers, avoid the motor, UART, I2C, ADC, and
SWD assignments, and leave PA5 free from conflict with the on-board LED.

## D9 — Phase 3 R-estimation uses "Option 3": RUNTIME stationarity-based R estimation + Flash-backed persistence

R-estimation uses runtime stationarity-based R estimation (not a one-time bake-in
constant). Measured R and gyro bias are persisted across reboots via an internal
Flash last page at **0x0801FC00** (1 KB, free on STM32F103RBTx 128 KB part).
Record layout: `{magic 0x4B414C31, float R, float bias}`. Rationale: keeps
measured params across reboots without re-flashing; user selected on 2026-06-24.

## D10 — Kalman measurement-noise R is the variance of the ACCEL ANGLE (deg^2), NOT gyro-rate variance

The Kalman measurement-noise R is the variance of the **accel angle** (deg²), not
the gyro-rate variance. This corrects the earlier plan wording "stationary gyro
samples -> variance = R": in the Lauszus 2-state filter the measurement is
`accel_angle`, so R must be its variance. Stationarity **detection** still gates
on the gyro (`|omega_dps| < 1.0 dps = at rest`), but the accumulated variance is
of `accel_angle`. User selected the physically-correct accel-angle variance on
2026-06-24.

## D11 — Two new Bluetooth AT commands on USART2: AT+SAVE and AT+GET

Two new AT commands on USART2:
- **AT+SAVE** persists current R/bias to Flash.
- **AT+GET** returns each parameter (R, BIAS, YAW) over USART2 as scaled integers.

Constraint: AT+SAVE only sets a volatile request flag inside the RX ISR; the actual
Flash erase/program runs in the main loop (Flash writes must never run in the
RX-complete ISR). Numeric output uses scaled integers because newlib-nano has no
float printf.

## D12 — Robustness (review fix): runtime variance floor + ParamStore Load guard

Runtime variance is clamped to a **1e-6f floor** before `Kalman_SetR`, and
`ParamStore_Load` rejects a stored R that is non-finite or `<= 0`, falling back
to Kalman defaults — prevents a degenerate near-zero/negative variance from
NaN-ing the filter or persisting across reboots.
