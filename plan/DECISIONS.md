# Decisions

Chronological decision log. Each entry records the choice, the reason, and the
evidence or source. Status terms per `.claude/rules/Edit_Workflow.md`.

## D1 — Architecture adopted from `plan.md` (module-per-driver + central FSM + fixed tick)

The architecture (one driver module per peripheral/algorithm, a central 7-state
FSM, and a fixed 10 ms control tick orchestrating Kalman → PID → S-curve → FSM)
is taken directly from the user's `plan.md`. It is not invented here.

A light benchmark of comparable open-source STM32 robot-car firmware confirms this
is a standard, well-supported shape — adopted as-is, no new structure borrowed:

- [TelecomParistoc/MotorController](https://github.com/TelecomParistoc/MotorController)
  — separates high-level order computation from a PID loop fed by coding-wheel +
  IMU feedback. Confirms the encoder + IMU + per-wheel-PID fixed-tick pattern.
- [SS2867/STM32RobotCar](https://github.com/SS2867/STM32RobotCar) — integrated
  line-tracking / parking / control project organized module-per-capability.
- [sametoguten/STM32-Line-Follower-with-PID](https://github.com/sametoguten/STM32-Line-Follower-with-PID)
  — PID line follower on STM32F103, confirms the centroid-error PID approach.
- [KevinAljas/vl53l1x-stm32-hal-driver](https://gitlab.com/KevinAljas/vl53l1x-stm32-hal-driver)
  and [pololu/vl53l1x-arduino ContinuousMultipleSensors](https://github.com/pololu/vl53l1x-arduino/blob/master/examples/ContinuousMultipleSensors/ContinuousMultipleSensors.ino)
  — confirm the XSHUT-gated, one-at-a-time I2C address-reassignment sequence used
  in [[ARCHITECTURE]] §vl53l1x.

## D2 — MCU is STM32F103RB; `plan.md` "SRAM 20 KB" was correct (D2 reversed)

`plan.md` listed the MCU as `F103RE` but stated `SRAM 20 KB`. These are
inconsistent: F103RE = 64 KB SRAM / 512 KB Flash, while 20 KB SRAM corresponds to
F103R**B** (128 KB Flash).

**Previously concluded (2026-06-23):** the chip was taken to be **F103RE**, treating
the "20 KB" figure as a typo.

**Reversed (2026-06-24):** user confirmed the physical board is the standard
**Nucleo-F103RB** = **STM32F103RBT6** (128 KB Flash / 20 KB SRAM). The original
"20 KB SRAM" figure was accurate; the chip is **F103RB**, not F103RE.

- CubeMX device selection, linker script, and startup file target **STM32F103RB**.
- The no-`malloc` / static-buffer discipline ([[OVERVIEW]] §Constraints) is kept
  regardless of the smaller SRAM — good practice, not a capacity workaround.
- Nucleo-F103RB is a real standard ST board (resolves the prior "no standard
  NUCLEO-F103RE" concern).

## D3 — Debug output reuses USART2 during bench tuning

`plan.md` line 354 proposed debug printing over "UART1 (PA9/PA10) or SWO". All of
these collide with assigned functions:

- USART1 default PA9/PA10 = LR-forward / LF-reverse motor inputs
- USART1 remap PB6/PB7 = I2C1 (SCL/SDA)
- SWO/TRACESWO (PB3) = LF-forward motor input

Decision (user-confirmed, 2026-06-23): during bench bring-up and tuning, before
HC-06 is physically attached, reuse **USART2 (PA2/PA3)** as the debug output.
Phases that depend on a debug path — Phase 3 (yaw via UART) and the Kalman /
straight-line data collection in [[REVIEW]] — run on USART2 with HC-06 detached.
Once HC-06 occupies USART2, on-target debug printing is unavailable; rely on LCD
status and pre-collected data instead.

## D4 — HAL + LL split per AN5044

- **HAL**: motor TIM PWM, I2C1, ADC1-DMA, USART2-DMA, SysTick.
- **LL**: Hall-sensor and shock-sensor EXTI initialization and IRQ handling.

Mixed HAL/LL in one project remains supported per ST application note AN5044,
and the latency-critical EXTI path remains LL as originally planned. CubeMX
selects the driver per generated initialization function, so `MX_GPIO_Init`
uses LL for direct GPIO/EXTI initialization while the hand-written motor runtime
API uses HAL TIM PWM start and compare writes.

## D5 — CubeMX-first, generated by Claude headless; hand-code in USER CODE sections

Each phase first configures the needed peripherals in STM32CubeMX and regenerates
code, then driver/algorithm logic is written by hand inside
`/* USER CODE BEGIN */ … /* USER CODE END */` blocks so CubeMX regeneration does
not overwrite it.

**Ownership (user-confirmed, 2026-06-23):** Claude performs the CubeMX generation
itself — it is *not* a manual user step. STM32CubeMX **6.17.0** at
`/opt/st/stm32cubemx_6.17.0/STM32CubeMX` (bundled JRE `/opt/st/jre`) does headless
CMake generation. **Verified 2026-06-23** by a smoke run of this script
(`STM32CubeMX -q <script>`), which produced `CMakeLists.txt`, `CMakePresets.json`,
`STM32F103XX_FLASH.ld`, and `startup_stm32f103xb.s`:

```
load STM32F103RBTx        # full part string required (not "STM32F103RB")
project name <name>
project toolchain CMake
project path <dir>
project generate
exit
```

The generated `startup_stm32f103xb.s` + `F103XX_FLASH.ld` independently confirm
the xB density (128 KB/20 KB) from §D2.

**Per-phase pattern:** the CubeMX CLI cannot mux pins/peripherals via script —
scripting only loads, sets project metadata, and generates. So Phase 0 creates a
baseline `.ioc` via `load`, and each later phase edits that `.ioc` (plain
key=value text) to add the phase's peripherals, then re-runs the headless
generate. Driver logic stays in `/* USER CODE BEGIN/END */` blocks. The user's
only role in the CubeMX step is the physical hardware gate. This is not full
end-to-end autonomous generation — coding still happens per phase via the
orchestration protocol.

> Phase 0 detail to confirm: **RESOLVED** (2026-06-24). MainLocation=Src, actual
> paths are `Src/`+`Inc/`, and [[ARCHITECTURE]] now matches.

## D7 — Drive hardware is a two-input H-bridge shield, not continuous-rotation servos

**Corrected 2026-06-24 from the physical wiring:** each DC motor uses two shield
inputs. The first input is forward, the second is reverse, and both LOW or both
PWM compare values 0 is stop/coast. Phase 1 therefore uses signed PWM on the
direction inputs and removes the original TIM1 servo-PWM assumption.

| Motor | Shield inputs | MCU PWM pins |
|---|---|---|
| Right rear | 21 / 22 | PB8 TIM4_CH3 / PB9 TIM4_CH4 |
| Right front | 17 / 16 | PB4 TIM3_CH1 / PB5 TIM3_CH2 |
| Left rear | 12 / 27 | PA9 TIM1_CH2 / PB10 TIM2_CH3 |
| Left front | 25 / 26 | PB3 TIM2_CH2 / PA10 TIM1_CH3 |

This mapping is the same polarity used by the previously working Smart_Car
firmware. Signed duty writes keep one direction compare non-zero and the opposite
compare at 0. PB3, PB4, PB8, PB9, and PB10 were previously assigned to
Hall/shock/XSHUT signals, so those later-phase signals must be remapped before
their phases. The signed-duty PWM path is the speed-modulation mechanism for
later closed-loop PID tuning.

## D8 — Resolve motor-pin conflicts on unused Nucleo header pins

**Decided 2026-06-24:** use PA4 for Hall RR, PA6 for shock EXTI, and
PA1/PA7/PA8 for Front/Left/Right VL53L1X XSHUT. These pins are exposed on the
NUCLEO-F103RB Arduino or ST morpho headers, avoid the motor, UART, I2C, ADC, and
SWD assignments, and leave PA5 free from conflict with the on-board LED.

## D9 — Use HSI-derived 64 MHz clock; retune PWM periods

**Decided and verified 2026-06-26:** the board setup has no HSE source available,
so the project uses HSI/2 into PLL ×16 for a 64 MHz SYSCLK. APB1 runs at 32 MHz
with APB1 timers at 64 MHz; APB2 timers also run at 64 MHz. ADC uses PCLK2/6.

Motor PWM keeps 20 kHz by using prescaler 0 and ARR 3199 on TIM1-TIM4. The motor
driver clamps signed duty to 3199. Build and ST-Link flash verification passed,
and the UART one-shot motor test emitted the expected sequence before user
observation confirmed all four motors move forward and reverse correctly.

## D10 — Drop hardware I2C1; bit-bang software I2C on PB6/PB7

**Root cause found 2026-06-26 (hardware-confirmed):** RF-reverse PWM (PB5 =
TIM3_CH2) was stuck HIGH — the right-front motor only spun in reverse. The cause
is **`I2C1_SMBA`, whose fixed pin on STM32F103 is PB5**: enabling the I2C1
peripheral couples SMBA onto the PB5 pad at the AF mux and overrides TIM3_CH2,
even though every TIM/AFIO/GPIO register reads correct. It is **not** a hardware
fault and **not** a HAL bug — verified by: PB5 toggles cleanly as plain GPIO; an
LL clean-room reconfig reproduced the stall; TIM3_CH2 works perfectly on its
no-remap pin PA7; and removing I2C1 frees PB5. (See stm32cubemx skill — "a
peripheral's secondary signal can steal a pin.")

**Decision:** keep all four motors on their current pins (RF stays PB4/PB5) and
**drop the hardware I2C1 peripheral entirely**, replacing it with a **software
bit-bang I2C master** on the existing device pins **SCL = PB6, SDA = PB7**
(open-drain, external pull-ups). Rationale — every hardware I2C placement collides
on this stack: I2C1 default PB6/PB7 drags SMBA onto PB5; I2C1 remap PB8/PB9 = RR
motor (TIM4_CH3/CH4); I2C2 PB10/PB11 = LR-reverse (TIM2_CH3). Software I2C avoids
all of them with no motor pin moves and serves the shared bus (MPU-6050 now; LCD,
VL53L1X ×3 later). Cost: `mpu6050.c` and `lcd.c` call a `soft_i2c` module instead
of `HAL_I2C_*`. **Status: implemented; pending hardware re-verification of RF
reverse + IMU/LCD on the soft bus.**

## D11 — Source correction: encoder API/units; Phase 4 speed telemetry added

**Corrected 2026-06-29 from the actual code.** [[ARCHITECTURE]] §encoder
previously documented a non-existent `Encoder_Update` returning `m/s` and a
"pulse period" formula. The real `encoder.{h,c}` API is:

- `Encoder_OnPulse(wheel)` (LL EXTI Hall ISR) increments `volatile pulse_count`.
- `Encoder_Sample(dt)` (called each `SPEED_PERIOD_MS` in the speed loop) computes
  `speed_rpm = (Δpulses / ENCODER_COUNTS_PER_REV) / dt · 60`.
- `Encoder_GetSpeed(wheel)` returns **RPM** (not m/s); the speed PID and FSM
  setpoints (`FSM_DRIVE_RPM`) are consistently in RPM.

`ENCODER_COUNTS_PER_REV` is a placeholder `20.0f` (Hall magnets × 2, both edges);
it must be validated on hardware before the Phase 4 `< 5%` number is trusted.

To enable that Phase 4 measurement, a new `Encoder_GetCount(wheel)` accessor and a
per-wheel USART2 telemetry line were added (`SPD=`/`CNT=`, see [[REVIEW]] §Phase 4
bench tuning). No control logic, gains, or encoder math changed.

## D12 — Phase 5 uses yaw-PID setpoint steering plus gyro-integrated turn completion

**Decided and implemented 2026-06-29 (commit 8e4f2ef).** Phase 5 stays software-only:
no CubeMX regeneration or new hardware gate was needed. Straight and turn behavior
is implemented inside the central FSM and feeds the existing per-wheel speed PID
loop through RPM setpoints instead of writing motor duty directly.

- Straight mode captures the current yaw on `STRAIGHT` entry, arms a jerk-limited
  `SCurve` ramp to `FSM_DRIVE_RPM`, and runs `pid_yaw` as an outer loop. Its
  correction offsets left/right wheel RPM targets (`base - corr`, `base + corr`),
  while the existing speed PID loop remains the inner actuator loop.
- Turn mode applies the requested pivot direction, integrates `fabsf(omega_dps) * dt`,
  stops at the placeholder `FSM_TURN_TARGET_DEG` of 90°, then returns to `STRAIGHT`
  and re-captures yaw.
- `FSM_Dispatch(yaw, omega_dps, dt)` is now called after `MPU6050_Read` and
  `Kalman_Update`, so Phase 5 receives fresh yaw/gyro data. The speed loop still
  consumes `FSM_GetSpeedSetpoint(wheel)` and applies `Motor_SetDuty` through the
  per-wheel speed PID.
- `scurve.{h,c}` is a small standalone module; `SCurve_Init(sc, target, accel_max,
  jerk)` records absolute acceleration and jerk limits, and `SCurve_Update(sc, dt)`
  advances ACCEL→CRUISE or DECEL→DONE without dynamic allocation.

All yaw gains, the turn target, ramp acceleration, jerk, and yaw-correction sign
remain tuning placeholders until hardware measurement. Build verification passed;
hardware verification is still pending.

## D13 — Phase 7 line tracing uses ADC-DMA normalized error into FSM setpoint steering

**Decided and implemented 2026-06-29 (working tree).** Phase 7 adds a small
`line_sensor.{h,c}` module around ADC1 CH0 on PA0 with circular DMA. The module
keeps an 8-sample static buffer, exposes a `LineSensorSample` snapshot with raw,
normalized, error, fresh, and active fields, and treats `error = normalized - center`
as the single-sensor line offset placeholder. `LineSensorConfig` carries raw min,
raw max, center, and invert calibration fields for later hardware tuning.

`LINE_TRACE` is now an active FSM state instead of a fail-safe stub. `AT+LINE`
enters the state. `FSM_Dispatch` receives the latest line error and active flag
from `main`; when the line sensor is inactive, the state clears its PID and speed
targets and keeps motion disabled. When active, `pid_line` offsets left/right RPM
setpoints around `FSM_DRIVE_RPM`, using the same inner per-wheel speed PID loop as
STRAIGHT and TURN.

ADC1 continuous conversion is enabled so `HAL_ADC_Start_DMA` refreshes the circular
buffer without polling. The HAL ADC half/full complete callbacks are owned by
`line_sensor.c` and guarded to `hadc1`.

Line PID gains, sensor raw range, center, invert setting, and steering sign remain
tuning placeholders. Build verification passed with `cmake --build build/Debug
--clean-first`; hardware line-follow verification is still pending.
